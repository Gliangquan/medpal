# Issue: MedPal 毕业设计版用户体验闭环整改计划

## 定位

本轮整改按“毕业设计可演示、可答辩、可完整走通核心场景”的目标推进，不追求商业平台级能力。

重点不是继续堆功能，而是把现有三端中用户最容易感知的断点补成闭环，让老师和答辩现场看到：

- 用户能顺利预约陪诊
- 陪诊员能顺利接单和完成服务
- 用户能确认完成并评价
- 评价后陪诊员星级能变化
- 电子病历能自动形成并可查看
- 个人设置不再是摆设，至少能改密码、控制病历可见性

## 本轮范围

### 一、本轮必须完成的闭环

1. 预约陪诊闭环优化
   - 限制预约开始时间范围，避免不现实的超长跨度选择
   - 根据陪诊时长自动计算结束时间，并在页面明确展示
   - 草稿保存后再次编辑不再重复新增草稿
   - 草稿恢复时把预约时间、陪诊员、联系方式等信息真正恢复回来

2. 订单完成闭环优化
   - 陪诊员点击“完成订单”后进入“待用户确认完成”
   - 用户在订单详情页可以真实看到并点击“确认完成”

3. 评价闭环优化
   - 用户提交评价后，陪诊员星级立即同步更新
   - 不再出现“评价成功但星级不变化”的体验

4. 个人设置闭环优化
   - 设置页补齐“修改密码”最小可用能力
   - 病历信息可见性不再只是开关展示，要参与实际权限控制
   - 清除缓存保持当前已修复策略，不强制退出登录

5. 电子病历闭环优化
   - 用户确认订单完成后，系统自动生成一份基础电子病历
   - 患者可在“我的病历”查看
   - 当患者开启“病历信息可见性”后，已接单陪诊员可按规则查看

### 二、本轮明确不做商业化扩展

以下功能保留为毕业设计“后续可扩展方向”，本轮不作为核心修复目标：

- 支付系统深度完善
- 财务结算后台
- 推荐系统
- 社区论坛/点赞/金牌陪诊员
- 陪诊员培训学习体系
- 平台级运营消息中心

## 用户视角下的整改原则

1. 先保证主链路走通，再考虑展示丰富度
2. 页面上出现的按钮必须能用，不能只是占位
3. 设置项必须对真实业务产生影响，不能“能保存但无效果”
4. 答辩时重点展示“预约—接单—完成—评价—病历—设置”全链路

## 执行方案

### Phase 1：预约与草稿
- [ ] 限制预约开始时间为“当前时间之后，且只开放合理预约窗口”
- [ ] 保留自动结束时间展示
- [ ] 修复草稿恢复时间字段不一致问题
- [ ] 修复重复保存草稿不断新增的问题

### Phase 2：订单与评价
- [ ] 修复订单详情页用户确认完成按钮判断错误
- [ ] 保证陪诊员发起完成后，患者端详情页可确认
- [ ] 修复评价提交后星级不更新问题

### Phase 3：设置与病历
- [ ] 新增设置页修改密码表单与接口
- [ ] 让病历可见性参与病历查询权限
- [ ] 用户确认订单完成时自动生成基础病历

## 验收标准

### 用户端
- [ ] 预约页只能选择合理时间范围内的开始时间
- [ ] 选择陪诊时长后可立即看到结束时间
- [ ] 草稿保存后再次进入能正确恢复
- [ ] 订单详情页在“待确认完成”状态下，患者可点击确认完成
- [ ] 用户评价后，陪诊员详情页星级可变化
- [ ] 设置页可以修改密码
- [ ] 患者开启病历可见性后，已接单陪诊员可查看病历；关闭后不可查看
- [ ] 订单完成后能自动生成至少一条基础病历记录

### 陪诊员端
- [ ] 接单后不能自己直接把订单变成已完成，只能发起完成申请
- [ ] 可在患者开放病历权限时查看其病历

## 风险说明

1. 电子病历自动生成属于毕业设计演示闭环，生成的是“基础病历草稿”，不是医院真实 EMR。
2. 病历权限按“是否为已接单/已服务陪诊员 + 患者是否开启可见性”控制，满足演示与论文需求即可。
3. 陪诊员主数据目前仍有 `user` / `companion` 双轨遗留问题，本轮先不做大重构。
